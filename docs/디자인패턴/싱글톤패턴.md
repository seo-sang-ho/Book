
## 디자인 패턴
프로그램을 설계할 때 발생한 문제를 객체 간의 상호 관계등을 이용해 해결 할 수 있도록 "규약" 형태를 만들어 놓은 것.

### 왜 사용하는가?

- 코드의 **재사용성 증가**
- 유지보수 용이
- 협업 시 **일관된 구조 유지**
- 검증된 설계 방식 사용으로 안정성 확보

### 대표적인 패턴 종류

- 생성 패턴: 객체 생성 방식 (싱글톤, 팩토리 등)
- 구조 패턴: 클래스/객체 구조 (어댑터, 데코레이터 등)
- 행위 패턴: 객체 간 상호작용 (옵저버, 전략 패턴 등)

## 싱글톤 패턴
- 하나의 클래스에 오직 하나의 인스턴스만 가지는 패턴

### 언제 사용하는가?

- 애플리케이션 전체에서 **하나만 존재해야 하는 객체**
  - 설정 정보 관리 객체
  - DB 커넥션 관리 객체
  - 로그 관리 객체

### 주의할 점

- TDD를 할 때 걸림돌이 된다. 각 테스트 마다 '독립적인' 인스턴스를 만들기가 어렵다.
이때 의존성 주입을 통해 모듈간의 결합을 느슨하게 만들어 해결 할 수 있다.

- 멀티스레드 환경에서 동시성 문제 발생 가능
  - Lazy Initialization 시 **동기화 처리 필요**

### Spring에서의 싱글톤

- Spring Bean은 기본적으로 **싱글톤 스코프**
- 개발자가 직접 싱글톤을 구현하기보다  
  Spring 컨테이너가 객체 생명주기를 관리

---

## 정리

- 프레임워크는 IoC를 통해 전체 흐름을 관리
- 디자인 패턴은 문제 해결을 위한 검증된 설계 방식
- 싱글톤은 유용하지만 테스트와 결합도 문제를 항상 고려해야 함
